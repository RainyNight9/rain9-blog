# 5.4 持续学习、模型迭代与 A/B 测试

在 AI 产品的生命周期中，开发并部署 1.0 版本仅仅是起步。AI 应用不是一个静态的代码集合，而是一个需要随着数据积累不断进化的“活体系统”。

如果说前几章我们关注的是“如何让 AI Agent 跑起来”，那么本章我们要解决的核心问题是：“如何让 AI Agent 越来越聪明？” 本节将结合我们 ai-data-analyzer 的架构，探讨如何建立数据飞轮、管理 Prompt 版本、进行安全的 A/B 测试，以及最终走向模型微调 (Fine-tuning)。

# 1. 打通数据飞轮 (Data Flywheel) 的闭环

我们在上一节 AnalysisController.submitFeedback 接口中已经预留了收集用户反馈 (Upvote/Downvote) 的入口。要建立真正的数据飞轮，我们需要完成以下完整的闭环：

  1. 收集反馈：用户点击“踩”，并留下期望的正确分析结果（Comment）。
  2. 数据沉淀：后端将 (原始数据, 用户 Prompt, 错误的 AI 回答, 用户期望的回答) 作为一个整体存入数据库（如 PostgreSQL）。
  3. 人工复核 (Human-in-the-loop)：工程师定期捞取这些 Bad Cases，分析是 Prompt 的问题，还是模型能力的问题。
  4. 迭代与验证：如果是 Prompt 问题，则更新 Prompt 并发布新版本；如果是模型能力瓶颈，则将这些修正后的数据对整理成高质量的训练集。
  5. 部署新模型/Prompt：带来更好的分析体验，从而吸引更多用户，产生更多数据。

# 2. 告别硬编码：Prompt 的版本控制与动态下发

在我们的初始代码 AnalysisService 中，你会发现 AGENT_SYSTEM_PROMPT 常常是直接以 const 字符串的形式硬编码在文件顶部的：

// ❌ 典型的反面教材：硬编码的 Prompt
const AGENT_SYSTEM_PROMPT = `你是一个专业的数据分析 AI Agent...`;

这种做法在生产环境中是致命的：

  • 无法热更新：每次修改 Prompt 哪怕只是改个标点，都需要重新编译、打包、重启整个后端服务。
  • 无法回滚：如果新 Prompt 导致了线上大面积幻觉，很难在一秒钟内切回老版本。
  • 缺乏协作：Prompt 工程师和开发人员的职责耦合在了一起。

# 实战改造：动态拉取 Prompt 配置

优秀的 AI 应用架构应当将 Prompt 视为独立于代码的配置资产。我们可以通过 Redis 或专门的 Prompt CMS (如 Langfuse, LangSmith) 来管理。

我们可以改造 AnalysisService，使其在运行时动态获取当前激活的 Prompt 版本：

// ✅ 好的做法：将 Prompt 与代码解耦，支持热更新
import { Injectable, Inject } from '@nestjs/common';
import { CACHE_MANAGER } from '@nestjs/cache-manager';
import type { Cache } from 'cache-manager';

@Injectable()
export class PromptManagerService {
  constructor(@Inject(CACHE_MANAGER) private cacheManager: Cache) {}

  async getActiveAgentPrompt(): Promise<string> {
    // 优先从 Redis 缓存中获取当前激活的 Prompt
    const cachedPrompt = await this.cacheManager.get<string>('config:prompt:agent_v_active');
    
    if (cachedPrompt) return cachedPrompt;

    // 如果缓存没有，则从数据库读取（这里作为示例直接返回 fallback）
    const fallbackPrompt = `你是一个专业的数据分析 AI Agent...`;
    await this.cacheManager.set('config:prompt:agent_v_active', fallbackPrompt, 3600); // 缓存 1 小时
    
    return fallbackPrompt;
  }
}

这样，当我们需要更新提示词时，只需更新数据库并刷新 Redis 缓存，所有的线上 Agent 实例将在下一次请求时立即使用新的“大脑”，实现真正的 0 停机热更。

# 3. 稳健前行：针对 LLM 的 A/B 测试路由

当你写了一个自认为更好的新 Prompt（版本 v2），或者想测试一个更便宜、速度更快的模型（比如想从 gpt-4o 降级到 gpt-4o-mini 以节省成本），千万不要直接全量替换！

AI 的非确定性意味着，你在测试集上的完美表现，可能在真实用户的各种长尾奇葩输入下全线崩溃。

基于特征的灰度发布与 A/B 测试：

我们可以在 AnalysisController 层或专门的路由拦截器中，实现流量拆分逻辑：

// backend/src/analysis/analysis.controller.ts (A/B 测试改造示例)

@Post('structured')
async analyzeStructured(@Body() payload: { prompt: string, userId: string }) {
  // 基于用户 ID 进行稳定的哈希路由，确保同一个用户始终命中同一个实验组
  const userHash = crypto.createHash('md5').update(payload.userId).digest('hex');
  const hashInt = parseInt(userHash.substring(0, 8), 16);
  const isExperimentGroup = hashInt % 100 < 20; // 20% 的流量进入实验组

  let result;
  
  if (isExperimentGroup) {
    Logger.log(`[A/B Test] User ${payload.userId} 命中实验组 (Prompt V2 + GPT-4o-mini)`);
    // 传递标识，让 Service 使用实验版 Prompt 和模型
    result = await this.analysisService.analyzeWithStructuredOutput(payload.prompt, { useExperiment: true });
  } else {
    Logger.log(`[A/B Test] User ${payload.userId} 命中对照组 (Prompt V1 + GPT-4o)`);
    // 维持原有稳定逻辑
    result = await this.analysisService.analyzeWithStructuredOutput(payload.prompt, { useExperiment: false });
  }

  return { success: true, data: result };
}

通过对比 A/B 两组在生产环境的 结构化解析失败率、用户点赞率 以及 平均 Token 消耗，我们可以用坚实的数据证明：新方案是否真正值得全量推行。

# 4. 突破瓶颈：何时以及如何进行模型微调 (Fine-tuning)

在实践中你最终会发现：无论你怎么打磨 Prompt，在处理你们公司特有的业务黑话、极其复杂的私有 JSON 结构、或者需要严格统一的输出语气时，单纯依赖 Prompt 不仅效果不稳定，还会因为 Prompt 过长导致巨额的 Token 成本和延迟。

这就是需要引入微调 (Fine-tuning) 的时刻。

# 实战：准备微调数据集 (JSONL)

微调并不是魔法，它的核心在于你积累的高质量数据（还记得我们第一节说的闭环吗？）。对于 OpenAI 的 Chat 模型，你需要准备符合特定格式的 .jsonl 文件。

我们可以编写一个脚本，将数据库中积累的几百条“人工复核修正后的最佳问答对”导出：

// fine_tuning_data.jsonl 示例格式
{"messages": [{"role": "system", "content": "你是一个专业的数据分析专家，只输出 JSON。"}, {"role": "user", "content": "分析这组电商数据..."}, {"role": "assistant", "content": "{\"summary\": \"大促期间客单价显著提升...\", \"confidenceScore\": 0.98}"}]}
{"messages": [{"role": "system", "content": "你是一个专业的数据分析专家，只输出 JSON。"}, {"role": "user", "content": "帮我看看这个库存预警表"}, {"role": "assistant", "content": "{\"summary\": \"发现3个SKU低于安全水位...\", \"confidenceScore\": 0.99}"}]}

通过 OpenAI 官方提供的 Fine-tuning API 或控制台提交这个文件，你将得到一个专属的模型 ID（例如 ft:gpt-4o-mini-0613:my-company:analysis-v1:8xT...）。

最后，只需将 ai-data-analyzer/backend/.env 中的环境变量替换：

# 将原先的基础模型
# OPENAI_MODEL=gpt-4o-mini
# 替换为你微调后的专属模型
OPENAI_MODEL=ft:gpt-4o-mini-0613:my-company:analysis-v1:8xT...

至此，你的 AI Agent 就完成了从“通用学霸”到“业务领域专家”的华丽蜕变！

本章小结： 不要将 AI 应用当成传统软件来维护。引入动态 Prompt 管理摆脱发版依赖；利用 A/B 测试对未知的 AI 表现进行灰度验证；通过收集用户反馈打通数据飞轮，并最终通过微调打造企业的护城河。这就是一个资深 AI 架构师必须要具备的长期演进思维。
