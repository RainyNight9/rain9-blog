在这个现代框架、组件库满天飞，Utility-first CSS（原子化 CSS）大行其道的年代，这确实是个好问题。

咱们这帮写前端的，现在谁还正儿八经手写“原生” CSS 啊？ 布局是现成的。 响应式是框架包办的。 无障碍（Accessibility）据说是组件库自带的。

要是真有什么地方需要改样式，咱们也就是微调个变量，加个 utility class，或者覆盖一下组件的 token 完事。

所以……到底为啥还要花时间学 CSS？

说实话，我很长一段时间也是这么想的。 直到最近，为了搞无障碍（Accessibility），生活狠狠地给了我一巴掌，逼着我重新做人。

最近我接了个看似人畜无害的活儿： 把一个老组件升级一下，让它符合现代 WCAG 无障碍标准。

你知道的——更好的可访问性、更高的对比度、正确的焦点状态（focus states）。 这可不是什么锦上添花的功能，而是硬指标。我看待无障碍一直很严肃。

但在实际操作中，这意味着大量的 CSS 工作。 容器得大改，但视觉设计必须保持一模一样。

结果……我翻车了。翻得挺惨。😅

在某个时刻，我甚至居然冒出了这种念头： CSS 可能才是前端开发里最难的部分。 比 JavaScript 还复杂。

（好吧，上一个开发者显然也不太懂 CSS，留下的坑确实不少 😬）

# 我以前 CSS 贼溜的

搞笑的是，我以前可是自诩 CSS 高手的。

盒子模型？小意思。 布局技巧？家常便饭。 我甚至能徒手把一个元素垂直水平居中——都不用去 Google 查代码！😄

所以……到底发生了什么？

# 无障碍开发的“叫醒服务” ♿

这次任务就像一盆冷水浇醒了我。

真正让人头秃的不是颜色或字体，而是那些“隐形”的坑：

  • Focus 轮廓线（focus outlines）把布局撑炸了
  • 键盘导航一走，突然暴露了 DOM 顺序是乱的
  • 轮廓线只显示在组件的一半上
  • 一旦 :focus-visible 介入，原本的定位 hack 全崩了

光是修复键盘导航这一项，就暴露了原本的布局有多脆弱。

没啥高深的“黑科技”。 全都是 CSS 基础知识碰上了现实世界的硬约束。

那一刻我突然意识到： 我已经很久没有进行过真正的 CSS 实战训练了。

# CSS 慢慢退居二线

不知道你们有没有同感，反正这几年，CSS 感觉就像是个“二等公民”。

在大多数项目里，我们用：

  • 一个组件库
  • 一套设计系统
  • 一个帮你搞定布局和响应式的框架

我们所谓的“CSS 工作”通常意味着：

  • 给单个组件调调样式
  • 改几个变量
  • 覆盖一下看着不太顺眼的地方

手机端适配？开箱即用。 Grid 网格？早就解决了。 间距？用 Tokens。

然后就是 Tailwind。

别误会——我也用 Tailwind。用的还挺多。 它确实快，结果也可预测。配合 shadcn/ui 这种库，你甚至不需要设计就能搞出像模像样的 UI。

老实说，对于业余项目或者 Demo，我已经懒得折腾了。 直接扔给 LLM（大模型），让它生成个好看的布局——带 Tailwind 类名的那种——然后直接用。

零 CSS 纠结。零痛苦。

当然，偶尔也会出问题。 这里 z-index 层级不对，那里溢出隐藏有点怪。 但跟以前徒手设计和维护整套布局比起来，这都不叫事儿。

我甚至都不怎么装 Stylelint 了——尽管我以前还给它贡献过代码（这让我觉得自己更像个渣男了）。 如果一个组件只有五行 CSS，还校验个啥劲啊？

# 讽刺的是……CSS 其实正在蓬勃发展 ✨

这就是最讽刺的地方。

CSS 这几年其实进化得飞起。

看看我们现在有什么：

  • CSS 变量（好吧，不算新了，但绝对是基石）
  • 容器查询（Container queries）
  • :has() 选择器
  • 级联层（Cascade layers）
  • 原生嵌套（Native nesting）
  • 现代色彩空间（Modern color spaces）
  • 滚动驱动动画（Scroll-driven animations）

今天的 CSS 比以往任何时候都强大。

我关注这些吗？ 算是吧。但不深。

因为在日常工作中，我几乎用不上。 框架处理了，库封装了，系统屏蔽了。

# 代沟

有趣的是，看着现在的年轻开发者，我总感觉还是“老一辈”更懂 CSS。

我甚至不确定现在还有没有正经教 CSS 的地方。 大学？估计没有。 培训班（Bootcamps）？教两天 CSS，然后直奔 JavaScript 全栈去了。

（话说——现在还有培训班吗？😅）

这是对的方向吗？ 还是说，这只是自然进化的必然？

# 手擀面与 CSS

有时候我在想，我对 CSS 的怀念是不是就像老奶奶坚持手擀面一样。

确实——手擀的更好吃。 确实——更有成就感。

但我们真的有那个闲工夫吗？

框架、工具、抽象层的存在是有理由的。 它们让我们更快，减少错误，标准化工作流程。

但是，无障碍（Accessibility）可不管你那一套抽象。

公平地说，大多数现代框架和组件库确实提供了开箱即用的无障碍支持——合理的 ARIA 属性、键盘支持、焦点管理。

但那种无障碍通常仅限于“快乐路径”（Happy Path）。一旦你需要自定义布局、视觉微调，或者搞点非标准的交互，你就立马被打回原形——这时候救你的只有原生 CSS，以及你脑子里仅存的那点知识。

当东西坏掉的时候——焦点丢失、键盘导航错乱、布局流崩坏—— 没有哪个 Tailwind 类名或组件 Prop 能救你。

只有 CSS 知识能救你。

# 所以……是 CSS 变难了吗？

我不这么认为。

我觉得是我们停止学习它了。

而无障碍开发用一种残酷的方式揭露了这一点。

# 你呢？

你现在还经常手写 CSS 吗？ 你还会从零开始设计布局吗？

还是说，CSS 已经变成了某种“失传的秘术”（Dark Knowledge）—— 一种只有当事情搞砸了，你才会被迫重新拾起的技能？

也许，偶尔锻炼一下这块“肌肉”还是值得的—— 哪怕只是为了好玩。
