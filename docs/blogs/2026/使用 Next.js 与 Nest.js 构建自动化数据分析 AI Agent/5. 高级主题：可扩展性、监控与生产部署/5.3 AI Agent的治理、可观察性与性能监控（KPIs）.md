# 5.3 AI Agent的治理、可观察性与性能监控（KPIs）

在生产环境中，“闭着眼睛开飞机”是极其危险的。传统软件监控 CPU/内存，但对于 AI Agent，我们需要监控全新的维度：**可观察性 (Observability)**。

## 1. 为什么 AI 应用需要特殊监控？

传统 API 的响应是确定的，而大模型的响应是非确定的 (Nondeterministic)。
- Prompt 稍微变动，结果可能天差地别。
- 模型 API 提供商可能暗中降级或调整模型。
- Token 成本可能因为用户的恶意长文本输入而飙升。

## 2. 核心 AI 监控指标 (KPIs)

建立一个 Grafana 仪表盘或使用专门的 AI 监控平台 (如 LangSmith, Helicone)，关注以下指标：

1. **Token 使用量与成本 (Cost & Tokens)**：按用户、按模型、按天统计消耗。
2. **生成延迟 (Latency/TTFT)**：首个 Token 响应时间 (Time To First Token) 和总生成时间。
3. **成功/失败率 (Success Rate)**：模型 API 超时、返回格式错误导致解析失败的比例。
4. **用户采纳率 (User Acceptance Rate)**：用户对 AI 结果点赞/点踩的比例（我们在 4.5 节实现的反馈机制）。

## 3. Nest.js 中的日志与追踪实现

为了实现可观察性，我们需要在每个环节打上 `traceId`。

使用 `nestjs-pino` 或集成 `OpenTelemetry`，在请求进入网关时生成一个全局唯一的 ID，并将此 ID 贯穿数据库查询、预处理、以及大模型调用的全生命周期。

```typescript
// 在 AnalysisService 中记录详细的 Trace
async analyzeText(content: string, traceId: string) {
  const startTime = Date.now();
  
  this.logger.log(`[Trace: ${traceId}] 开始预处理数据`);
  const cleanData = this.preprocess(content);
  
  this.logger.log(`[Trace: ${traceId}] 发起大模型请求，长度: ${cleanData.length}`);
  const result = await this.llm.chat(cleanData);
  
  const duration = Date.now() - startTime;
  this.logger.log(`[Trace: ${traceId}] 任务完成，耗时 ${duration}ms`);
  
  // 异步将这些指标发送到监控系统
  this.metricsService.recordLatency('llm_call', duration);
  
  return result;
}
```

通过完备的监控，你能在用户投诉之前，就发现某个 Prompt 在最新的 GPT 模型上出现了退化，从而迅速修复。
