# 5.4 持续学习、模型迭代与A/B测试

AI 产品不是发布即完成，而是一个随着数据积累不断进化的“活体”。本节探讨如何利用用户反馈建立数据飞轮。

## 1. 建立数据飞轮 (Data Flywheel)

我们在 4.5 节收集了用户的反馈 (点赞 👍 / 踩 👎)。
当用户点了“踩”，我们应该允许用户输入预期的正确结果。这些“失败的案例 + 正期的结果”构成了极其宝贵的**高质量数据集**。

飞轮循环：
1. Agent 提供服务 -> 2. 收集负面反馈 -> 3. 工程师审查并修正 Prompt 或准备微调数据 -> 4. 更新 Agent -> 1. 提供更好的服务。

## 2. 提示词迭代与版本控制

提示词 (Prompt) 也是代码，必须被版本控制！
不要把 Prompt 硬编码写死在 Service 字符串里。建议将它们抽离到配置文件、数据库，或者专门的 Prompt 管理系统 (Prompt CMS) 中。

```typescript
// 好的做法：从配置中心拉取指定版本的 Prompt
const promptTemplate = await this.promptConfigService.get('data_analysis_v2.1');
const prompt = promptTemplate.replace('{data}', data);
```

## 3. 针对 AI 的 A/B 测试

当你写了一个自认为更好的新 Prompt，或者切换了一个更便宜的模型时，千万不要直接全量替换。

**使用 A/B 测试验证效果：**

在网关层或业务层进行流量拆分：
- **组 A (对照组 80%流量)**：使用旧的 Prompt v1 和 GPT-4o。
- **组 B (实验组 20%流量)**：使用新的 Prompt v2 和 Claude 3.5。

通过对比 A/B 两组在上一节提到的监控指标（尤其是**用户点赞率**和**任务成功率**），用数据来决定新版本是否真正带来了提升。

## 4. 模型微调 (Fine-tuning) 的时机

如果你发现，无论怎么调整 Prompt，大模型依然无法理解你们公司内部特有的黑话或特定的复杂数据结构，那么就到了考虑**微调 (Fine-tuning)** 的时候了。
将你收集到的几百条高质量问答对格式化为 JSONL，通过 OpenAI 的 Fine-tuning API 训练一个专属于你业务的定制模型。
