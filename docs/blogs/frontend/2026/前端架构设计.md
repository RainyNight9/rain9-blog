# 前端架构设计

## 1. 业务与体验目标

1. 业务边界：
   - 产品要解决什么问题？核心用户旅程有哪些（登录、浏览、下单、支付、内容发布等）？
   - 哪些页面/模块是核心链路，哪些是可延后或可降级的非核心链路？
2. 体验指标：
   - 首屏与交互体验：TTFB、FCP、LCP、INP、CLS 的目标值和衡量方式。
   - 弱网/低端机：加载策略、骨架屏、降级策略、离线可用性。
3. 可演进性与团队协作：
   - 项目生命周期（短期活动页 vs 长期产品）。
   - 团队规模与协作模式（多人并行、跨团队共建、组件复用诉求）。

## 2. 技术栈与运行环境

1. 运行环境与兼容：
   - 浏览器与版本范围、是否要支持 WebView、小程序、Electron、桌面端等。
   - 国际化/多时区/多语言输入，字体与排版差异。
2. 渲染形态选择：
   - CSR：交互强、迭代快，但首屏与 SEO 可能弱。
   - SSR：首屏更好，适合内容型/SEO，但成本更高。
   - SSG/ISR：适合内容与营销页，配合增量更新。
   - 混合渲染：按路由/页面选择最佳渲染策略。
3. 框架与工具链：
   - React/Vue/Svelte 等选择依据（生态、团队熟悉度、性能诉求）。
   - 包管理与构建（pnpm/yarn、Vite/Webpack、Babel/SWC）。

## 3. 模块划分与项目结构

1. 按领域划分模块：
   - 以业务域而非页面堆叠组织代码（如 user、order、payment、content）。
   - 领域模块内部再分：ui、model、service、hooks/utils。
2. 分层结构建议：
   - presentation：页面与视图组件（负责渲染与交互）。
   - domain：领域模型、状态与业务规则。
   - application：编排用例（跨领域流程、接口编排）。
   - infrastructure：HTTP/Storage/Telemetry 等基础设施封装。
3. 依赖方向与边界：
   - 让上层依赖下层，禁止反向依赖，避免组件互相引用形成“网状结构”。
   - 对外部依赖（SDK/第三方库）做适配层，避免业务代码直接散落调用。

## 4. 路由与导航体系

1. 路由设计：
   - 路由分组、动态路由、权限路由、嵌套路由、布局路由。
   - URL 作为状态：可分享、可回放、可 SEO 的关键参数应在 URL 中。
2. 访问控制：
   - 未登录跳转、登录态刷新、token 过期处理、第三方登录回调。
   - 路由级别的权限守卫与资源级别的鉴权分离。
3. 预加载与分包：
   - 路由级 code splitting、按业务域拆包、按场景预加载。
   - 首屏最小化依赖、次屏按需加载。

## 5. 状态管理与数据流

1. 状态分类：
   - Server State：来自后端的数据（列表、详情、权限、配置）。
   - Client UI State：弹窗、选中态、编辑态、表单草稿等。
   - Cross-Page State：跨页面共享的用户信息、购物车等。
2. 管理方案：
   - Server State：优先用请求库的缓存能力（如 Query 类库），减少手写缓存。
   - Client State：局部优先（组件 state），必要时再上全局状态。
   - 表单状态：表单库与 schema 校验（规则复用、错误提示统一）。
3. 数据一致性与刷新策略：
   - 缓存键设计、失效策略、乐观更新与回滚。
   - 幂等与重复提交防护（按钮节流、请求幂等键）。

## 6. API 与通信层设计

1. API 契约：
   - 统一的响应结构、错误码规范、分页/排序/过滤统一参数。
   - OpenAPI/契约校验，减少前后端对齐成本。
2. 通信封装：
   - 统一的 request client（拦截器、超时、重试、取消、错误归一化）。
   - token 刷新策略：并发刷新锁、失败重登、无感续期策略。
3. 实时与流式：
   - WebSocket/SSE：用于消息、协作、流式输出。
   - 断线重连、心跳、消息顺序与幂等处理。

## 7. 性能与加载策略

1. 性能预算：
   - 按页面给出 JS/CSS/图片体积预算，设定 LCP/INP 目标。
2. 资源策略：
   - 图片：响应式图片、懒加载、WebP/AVIF、CDN。
   - 字体：子集化、font-display、预加载策略。
   - 第三方脚本：延迟加载、沙箱隔离、失败降级。
3. 代码与渲染优化：
   - 减少不必要渲染：memo、拆分组件、虚拟列表。
   - 重计算控制：节流/防抖、Web Worker、分帧处理。
   - 长任务治理：监控与拆分，避免卡顿。

## 8. 设计系统与组件体系

1. 组件分层：
   - 基础组件：Button/Input/Modal（无业务语义）。
   - 业务组件：OrderCard/UserAvatar（有业务语义，可复用）。
   - 页面组件：只服务于单页，不强求复用。
2. 设计系统：
   - 主题、token（颜色、间距、字体、阴影）、暗色模式。
   - 可访问性：键盘操作、ARIA、对比度、焦点管理。
3. 组件文档与测试：
   - Storybook/文档站，明确 API 与用法。
   - 组件回归测试与视觉回归测试。

## 9. 工程化与质量保障

1. 代码规范：
   - lint、格式化、提交规范、变更日志策略。
   - 单元边界与导入规则，避免跨层引用。
2. 测试策略：
   - 单元测试：工具函数、业务规则、关键 hooks。
   - 集成测试：关键用户旅程（登录-下单-支付）。
   - E2E：核心路径自动化回归。
3. CI/CD：
   - 构建、测试、产物发布、灰度、回滚。
   - 预览环境：PR 级别预览，减少联调成本。

## 10. 安全与风控

1. 前端安全基础：
   - XSS/CSRF/CSP、输入输出处理、Cookie 策略（SameSite/HttpOnly）。
   - 依赖安全：漏洞扫描、锁版本策略、供应链风险。
2. 鉴权与权限展示：
   - 前端权限只能用于展示与交互控制，真正权限必须由后端校验。
   - 防止越权：接口层统一处理 401/403，UI 层一致反馈。
3. 风控与反作弊：
   - 行为埋点、验证码、人机校验、频控策略、设备指纹（合规前提下）。

## 11. 可观测性：监控、日志与埋点

1. 性能监控：
   - Web Vitals 上报、首屏耗时、资源加载失败率、JS 错误率。
2. 日志与追踪：
   - 统一 requestId/traceId，前端把 traceId 透传给后端，便于排障。
   - 关键链路埋点：登录成功率、支付成功率、下单转化等。
3. 报警与回放：
   - 错误聚合、版本维度对比、用户会话回放用于复现问题。

## 12. 发布、灰度与多环境

1. 环境管理：
   - dev/test/stage/prod 环境隔离，配置按环境下发。
   - feature flag：新功能开关、灰度人群、快速回滚。
2. 发布策略：
   - 静态资源版本化与缓存策略（hash、Cache-Control、CDN 回源）。
   - 灰度发布：按用户、地域、Header、百分比进行流量切分。
3. 兼容与回滚：
   - 前后端版本兼容窗口，接口版本管理，避免“先发前端后端未就绪”。

## 13. 多端与微前端（可选）

1. 多端统一策略：
   - 组件与业务逻辑复用（monorepo、多包架构、共享 SDK）。
   - 适配层：Web/小程序/APP 的差异抽象。
2. 微前端适用场景：
   - 多团队、多业务线、独立发布诉求强。
   - 需要明确：运行时隔离、路由冲突、样式隔离、性能成本。
3. 与设计系统的配合：
   - 统一 UI token、统一基础组件，减少割裂感。

---

上面的内容可以直接当作「前端架构设计检查清单」：从业务目标与渲染策略 → 模块划分与数据流 → 性能与组件体系 → 工程化、可观测性与发布治理，逐项补齐后，就是一份可落地的前端架构方案。
